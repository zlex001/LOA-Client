# 构建与发布流程

本文档详细说明 LOA 客户端的构建流程、版本管理、热更新部署等内容。

## 构建类型

### 开发构建（Development Build）

**用途**：内部测试

**特点**：
- 包含调试信息（.pdb）
- 允许 Profiler 连接
- 日志输出详细
- 性能未优化

**输出**：
- APK/IPA/EXE
- game.dll.bytes + game.dll.pdb.bytes

### 发布构建（Release Build）

**用途**：正式发布

**特点**：
- 移除调试信息
- 代码和资源剥离优化
- IL2CPP 代码保护
- 包体积最小

**输出**：
- APK/IPA/EXE
- game.dll.bytes

## 版本管理

### 版本号规则

采用**语义化版本**：`主版本.次版本.修订版本`

**格式**：`major.minor.patch`

**示例**：
- `1.0.0`：首次发布
- `1.0.1`：热更新修复（无需重新提审）
- `1.1.0`：新增功能（热更新）
- `2.0.0`：大版本更新（需要重新提审）

### 版本号管理

#### 应用版本（AppVersion）

**位置**：`Assets/Launcher/Config.asset`

```
appVersion: "1.0.0"
```

**说明**：
- 打包到主包中
- 对应商店版本号
- 修改后需要重新打包提审

#### 热更新版本（HotVersion）

**位置**：`Assets/Launcher/Config.asset`

```
hotVersion: "1.0.0"
```

**说明**：
- 本地热更新版本号
- 用于对比服务器版本
- 热更新后自动更新

### 版本号更新时机

| 场景 | 更新版本 | 说明 |
|------|----------|------|
| 重新提审 | AppVersion + HotVersion | 主版本更新 |
| 热更新 | 仅 HotVersion | 无需提审 |
| Bug修复 | HotVersion 的 patch | 如 1.0.0 → 1.0.1 |
| 新增功能 | HotVersion 的 minor | 如 1.0.0 → 1.1.0 |

## 构建流程

### 步骤一：准备构建

#### 1. 检查代码

```bash
# 确保无编译错误
# 在 Unity 中查看 Console，确保无红色错误
```

#### 2. 更新版本号

编辑 `Assets/Launcher/Config.asset`：
```
appVersion: "1.0.0"
hotVersion: "1.0.0"
```

#### 3. 清理缓存

```bash
# 删除 Library 目录（可选，首次构建推荐）
rm -rf Library/

# 删除构建目录
rm -rf Build/
```

### 步骤二：构建热更新 DLL

#### Unity 菜单构建

1. 菜单：`HybridCLR → Build → BuildHotUpdate`
2. 等待构建完成
3. 输出：`Assets/StreamingAssets/game.dll.bytes`

#### 命令行构建

```bash
# Windows
Unity.exe -quit -batchmode -projectPath . -executeMethod HybridCLR.Editor.Commands.BuildHotUpdate

# macOS/Linux
Unity -quit -batchmode -projectPath . -executeMethod HybridCLR.Editor.Commands.BuildHotUpdate
```

### 步骤三：构建 AssetBundle

#### Unity 菜单构建

1. 菜单：`Assets → Build AssetBundles`
2. 选择目标平台（Android/iOS/Windows）
3. 等待构建完成
4. 输出：`Build/AssetBundles/[Platform]/`

#### 命令行构建

```bash
Unity -quit -batchmode -projectPath . -executeMethod BuildTools.BuildAssetBundles -platform Android
```

### 步骤四：构建应用

#### Unity 菜单构建

1. 菜单：`File → Build Settings`
2. 选择目标平台
3. 点击 "Build" 或 "Build and Run"
4. 选择输出目录

#### 命令行构建

```bash
# Android
Unity -quit -batchmode -projectPath . -buildTarget Android -executeMethod BuildTools.BuildAndroid

# iOS
Unity -quit -batchmode -projectPath . -buildTarget iOS -executeMethod BuildTools.BuildIOS

# Windows
Unity -quit -batchmode -projectPath . -buildTarget Win64 -executeMethod BuildTools.BuildWindows
```

## 热更新部署

### 步骤一：生成资源列表

```bash
# 在 Build/HotUpdate/ 目录下生成 res.txt
cd Build/HotUpdate/

# 格式：文件名|MD5|大小
echo "game.dll.bytes|$(md5 game.dll.bytes)|$(stat -f%z game.dll.bytes)" > res.txt
```

### 步骤二：上传到服务器

```bash
# 使用 rsync 或 scp 上传
rsync -avz Build/HotUpdate/ user@server:/var/www/hotupdate/

# 或使用 CDN 上传工具
cdn-upload Build/HotUpdate/ --bucket loa-hotupdate
```

### 步骤三：更新版本号

在服务器上更新版本号：

```bash
# 更新 /api/hot/version 返回的版本号
curl -X POST http://server:8880/api/admin/update-version -d '{"version":"1.0.1"}'
```

### 步骤四：验证热更新

1. 运行旧版本客户端
2. 观察版本检查流程
3. 确认下载并加载新版本
4. 测试新功能

## 发布流程

### 首次发布

1. **准备工作**：
   - 更新 AppVersion 和 HotVersion
   - 构建开发版本进行测试
   - 修复所有已知问题

2. **构建发布版本**：
   - 构建热更新 DLL
   - 构建 AssetBundle
   - 构建应用（Release 配置）

3. **提交审核**：
   - 上传到商店（Google Play / App Store）
   - 填写更新说明
   - 等待审核

4. **部署热更新资源**：
   - 上传热更新资源到服务器/CDN
   - 设置版本号

### 热更新发布

1. **准备工作**：
   - 更新 HotVersion（不改 AppVersion）
   - 测试新功能

2. **构建热更新资源**：
   - 构建热更新 DLL
   - 构建 AssetBundle
   - 生成资源列表

3. **部署**：
   - 上传到服务器/CDN
   - 更新服务器版本号

4. **验证**：
   - 运行旧版本客户端
   - 确认自动更新
   - 测试新功能

## 资源列表生成

### res.txt 格式

```
game.dll.bytes|a1b2c3d4e5f6|1024567
game.dll.pdb.bytes|b2c3d4e5f6a1|2048123
mscorlib.dll.bytes|c3d4e5f6a1b2|3072456
System.dll.bytes|d4e5f6a1b2c3|2560123
ui_home.bundle|e5f6a1b2c3d4|512345
resdepend.txt|f6a1b2c3d4e5|1024
```

**每行格式**：`文件名|MD5|大小（字节）`

### 生成脚本

```python
# tools/generate_res_list.py
import os
import hashlib

def get_md5(filepath):
    with open(filepath, 'rb') as f:
        return hashlib.md5(f.read()).hexdigest()

def get_size(filepath):
    return os.path.getsize(filepath)

def generate_res_list(directory, output):
    with open(output, 'w') as out:
        for filename in os.listdir(directory):
            filepath = os.path.join(directory, filename)
            if os.path.isfile(filepath):
                md5 = get_md5(filepath)
                size = get_size(filepath)
                out.write(f"{filename}|{md5}|{size}\n")
    
    print(f"Generated: {output}")

# 使用
generate_res_list('Build/HotUpdate', 'Build/HotUpdate/res.txt')
```

## 热更新部署

### CDN 部署

#### 1. 上传资源到 CDN

```bash
# 使用 AWS S3
aws s3 sync Build/HotUpdate/ s3://loa-hotupdate/android/

# 使用阿里云 OSS
ossutil cp -r Build/HotUpdate/ oss://loa-hotupdate/android/

# 使用腾讯云 COS
coscmd upload -r Build/HotUpdate/ loa-hotupdate/android/
```

#### 2. 配置 CDN 缓存

- 静态资源缓存：7天
- 版本文件缓存：30秒（快速更新）

#### 3. CDN URL 格式

```
https://cdn.example.com/hotupdate/{Platform}/{FileName}

示例：
https://cdn.example.com/hotupdate/android/game.dll.bytes
https://cdn.example.com/hotupdate/ios/game.dll.bytes
```

### 服务器部署

#### 1. 上传资源

```bash
# 使用 rsync
rsync -avz Build/HotUpdate/ user@server:/var/www/hotupdate/android/

# 使用 scp
scp -r Build/HotUpdate/* user@server:/var/www/hotupdate/android/
```

#### 2. 配置 Nginx

```nginx
server {
    listen 8880;
    server_name example.com;
    
    # 热更新资源
    location /android/ {
        alias /var/www/hotupdate/android/;
        
        # 允许跨域
        add_header Access-Control-Allow-Origin *;
        
        # 缓存控制
        expires 7d;
    }
    
    # 版本接口
    location /api/hot/version {
        proxy_pass http://localhost:8080;
    }
}
```

#### 3. 更新版本号

```bash
# 调用服务器API更新版本号
curl -X POST http://server:8880/api/admin/update-version \
    -H "Content-Type: application/json" \
    -d '{"version":"1.0.1"}'
```

## 性能优化

### 1. 代码剥离（Code Stripping）

```
Player Settings → Other Settings → Managed Stripping Level: High
```

**作用**：移除未使用的代码，减小包体积

### 2. 纹理压缩

- Android：使用 ASTC 格式
- iOS：使用 ASTC 或 PVRTC 格式
- Windows：使用 DXT5 格式

### 3. AssetBundle 压缩

```csharp
BuildAssetBundleOptions options = 
    BuildAssetBundleOptions.ChunkBasedCompression;  // 使用 LZ4 压缩
```

### 4. IL2CPP 优化

```
Player Settings → Configuration → Scripting Backend: IL2CPP
Player Settings → Configuration → C++ Compiler Configuration: Master
```

## 版本发布流程

### 主版本发布（需提审）

1. 更新 AppVersion 和 HotVersion
2. 构建完整应用
3. 提交商店审核
4. 审核通过后发布
5. 部署热更新资源

### 热更新发布（无需提审）

3. 部署到服务器/CDN
4. 更新版本号

## 回滚机制

### 热更新回滚

**场景**：热更新版本有严重 bug

**步骤**：
1. 恢复服务器上的旧版本资源
2. 更新版本号为旧版本号
3. 玩家自动回滚到旧版本

**注意**：
- 数据兼容性问题
- 回滚前备份当前版本

### 应用回滚

**场景**：主版本有严重 bug

**步骤**：
1. 在商店下架当前版本
2. 恢复旧版本
3. 修复 bug 后重新提审

## 常见问题

### Q: 构建时间过长？

A: 
- 启用增量构建
- 使用缓存服务器（Cache Server）
- 减少不必要的资源
- 使用 SSD 硬盘

### Q: 包体积过大？

A:
- 检查未使用的资源
- 启用代码剥离（Code Stripping）
- 压缩纹理和音频
- 使用 AssetBundle 分离资源

### Q: 热更新失败？

A:
- 检查服务器资源是否正确部署
- 检查版本号是否更新
- 检查资源列表（res.txt）格式
- 查看客户端日志（[Hot] 标签）

### Q: 如何测试热更新？

A:
1. 构建旧版本（如 1.0.0）并安装
2. 运行旧版本
3. 部署新版本热更新资源（1.0.1）
4. 重启应用，观察自动更新过程

## 总结

LOA 客户端构建与发布流程包括：

1. **版本管理**：语义化版本号，区分应用版本和热更新版本
2. **构建流程**：DLL 构建、AssetBundle 构建、应用构建
3. **热更新部署**：生成资源列表、上传到服务器、更新版本号
4. **发布检查**：构建前、构建后、发布前的完整检查清单
5. **平台适配**：Android、iOS、Windows 的特定配置
6. **回滚机制**：热更新回滚和应用回滚

遵循这些流程可以确保发布的稳定性和可靠性。
